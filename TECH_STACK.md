## Overview

This is a **monorepo** — one GitHub repository containing two separate applications:

```
bedr-admin/                  ← Root of your GitHub repo
├── frontend/                ← Next.js app (deployed to Vercel)
├── backend/                 ← Express.js app (deployed to Render)
└── README.md
```

They communicate over HTTP — the frontend calls the backend's REST API.

## Backend Stack

### Runtime & Framework
| Tool | Version | Why |
|------|---------|-----|
| Node.js | 20.x LTS | Stable, widely supported |
| Express.js | 4.x | The assignment requires it. Minimal, unopinionated HTTP framework |
| TypeScript | No | We use plain JavaScript (CommonJS) to keep things simple for learning |

### Database Layer
| Tool | Version | Why |
|------|---------|-----|
| Supabase | Cloud | Provides hosted PostgreSQL — the assignment requires this |
| Prisma ORM | 5.x | Type-safe DB client + migrations + readable schema definition |
| `@prisma/client` | auto | Generated from your schema, used in route handlers |

> **What is Prisma?** Instead of writing raw SQL like `SELECT * FROM beds WHERE id = $1`, you write `prisma.bed.findUnique({ where: { id } })`. It generates SQL for you, prevents injection attacks, and gives you a readable schema file.

> **What is an ORM?** Object-Relational Mapper. It maps database rows to JavaScript objects so you work with JS instead of SQL.

### Validation
| Tool | Why |
|------|-----|
| `express-validator` | Validates incoming request bodies and params. Ensures email is valid, strings aren't empty, etc. |

### Middleware
| Tool | Why |
|------|-----|
| `cors` | Allows the frontend (on a different origin) to call the backend API |
| `morgan` | HTTP request logger — shows every request in terminal for debugging |
| `helmet` | Sets security headers — good practice, minimal effort |
| `dotenv` | Loads environment variables from a `.env` file |

### Backend Folder Structure
```
backend/
├── prisma/
│   ├── schema.prisma        ← Your data model lives here
│   └── migrations/          ← Auto-generated by Prisma
├── src/
│   ├── index.js             ← App entry point, middleware setup
│   ├── routes/              ← One file per resource
│   │   ├── flats.js
│   │   ├── rooms.js
│   │   ├── beds.js
│   │   └── tenants.js
│   ├── controllers/         ← Business logic functions called by routes
│   │   ├── flats.controller.js
│   │   ├── rooms.controller.js
│   │   ├── beds.controller.js
│   │   └── tenants.controller.js
│   ├── middleware/
│   │   ├── errorHandler.js  ← Global error handler
│   │   └── validate.js      ← Reusable validation middleware
│   └── lib/
│       └── prisma.js        ← Prisma client singleton
├── .env                     ← Never commit this
├── .env.example             ← Commit this (shows what vars are needed)
├── package.json
└── .gitignore
```

> **Why separate routes and controllers?** Routes define *what URL does what*. Controllers define *what actually happens*. Keeping them separate makes each file short and readable. This is called the **MVC pattern** (Model-View-Controller) — you've seen it in FastAPI's router + service separation in finsight-backend.

## Frontend Stack

### Framework & Language
| Tool | Version | Why |
|------|---------|-----|
| Next.js | 14.x | Assignment requires it. Uses App Router |
| React | 18.x | Comes with Next.js |
| JavaScript | ES Modules | No TypeScript — keep it simple |

> **What is the App Router?** Next.js 14 has two routing systems. App Router (newer) uses `app/` directory and React Server Components. We'll use it because it's the current standard.

### Styling
| Tool | Why |
|------|-----|
| Tailwind CSS | Utility-first CSS. You write `className="flex gap-4 p-2 bg-white"` instead of separate CSS files. Fast to build with |
| shadcn/ui | Pre-built React components (buttons, tables, dialogs, etc.) built on Tailwind. You copy components into your project — no black-box library |
| clsx | 
| tailwind-merge |
|class-variance-authority |
| tailwindcss-animate |
| radix-ui |
| next-themes |
| sonner |
> **What is shadcn/ui?** Unlike a regular component library (like MUI), shadcn/ui copies the component source code INTO your project. You own it, can edit it, and it uses Tailwind under the hood. It's the current industry standard for Next.js projects.

### Data Fetching
| Tool | Why |
|------|-----|
| `axios` | HTTP client for calling the backend API. Cleaner than raw `fetch` with better error handling |
| TanStack Query (React Query) | Manages server state — caching, loading states, refetching. Much cleaner than `useEffect` + `useState` for data fetching |

> **What is TanStack Query?** Instead of writing `useEffect(() => { fetch(...).then(setData) }, [])` in every component, you write `useQuery({ queryKey: ['flats'], queryFn: getFlats })` and it handles loading, errors, caching, and refetching automatically.

## Database (Supabase + PostgreSQL)

| What | Detail |
|------|--------|
| Provider | Supabase (free tier) |
| Database | PostgreSQL 15 |
| Connection | Direct connection string via `DATABASE_URL` env var |
| Migrations | Managed by Prisma (`prisma migrate dev`) |
| Hosting | Supabase cloud (no local DB needed) |

> **What is Supabase?** It's a hosted PostgreSQL service with a nice dashboard. Think of it as Firebase but with a real relational database. You create a project, get a connection string, and Prisma connects to it.

## Environment Variables

### Backend `.env`
```
DATABASE_URL="postgresql://user:password@host:5432/dbname?pgbouncer=true"
PORT=5000
NODE_ENV=development
FRONTEND_URL=http://localhost:3000
```

### Frontend `.env.local`
```
NEXT_PUBLIC_API_URL=http://localhost:5000/api
```

> **Why `NEXT_PUBLIC_`?** In Next.js, env variables are server-side by default (not exposed to the browser). Prefixing with `NEXT_PUBLIC_` makes them available in the browser JavaScript. Since your API URL needs to be called from the browser, it needs this prefix.

## How the Pieces Connect

```
Browser (Next.js on Vercel)
      │
      │  HTTP requests to /api/...
      ▼
Express.js API (on Render)
      │
      │  Prisma ORM queries
      ▼
PostgreSQL (on Supabase)
```

1. User clicks "Create Flat" in the browser
2. Next.js frontend sends `POST /api/flats` to the Express backend
3. Express validates the body, calls the controller
4. Controller uses Prisma to insert a row into PostgreSQL on Supabase
5. Prisma returns the created row as a JS object
6. Express sends it back as JSON
7. TanStack Query receives the response, updates the UI automatically
